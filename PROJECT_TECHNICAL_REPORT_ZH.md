# Experiment 项目技术汇报报告

## 1. 项目定位与研究目标

`Experiment` 是一个面向大语言模型安全评测的实验系统，目标是围绕“攻击生成、越狱执行、防御插入、结果分析”构建一条完整、可复现、可扩展的实验链路。项目聚焦的问题不是单点问答能力，而是模型在复杂越狱场景中的安全性评估与对抗性研究。

从研究视角看，该项目主要回答四类问题：

1. 已有攻击提示经过改写后，是否能够形成更强的越狱样本。
2. 目标模型在单轮与多轮交互中，是否会逐步暴露出更高的安全脆弱性。
3. 输入层、交互层、输出层三类防御机制，分别能在什么位置发挥作用。
4. 如何将原始 JSONL 结果转化为结构化判定、统计表和图表，支持实验结论输出。

因此，这个项目本质上不是单一脚本集合，而是一个围绕 LLM 安全评测构建的模块化实验平台。

## 2. 总体架构与主链路

项目整体链路可概括为：

`Attack_Dataset -> Redteam -> Jailbreak -> Defense(可选) -> Analyze`

这条链路反映了系统的核心设计思想：先构造攻击，再执行攻击，过程中可插入防御，最终对结果做统一判定与统计。

在工程层面，项目分为五个主模块和两个公共支撑层：

1. `Attack_Dataset`
   负责存放标准化后的攻击数据集。
2. `Redteam`
   负责生成或改写越狱提示，扩充攻击样本空间。
3. `Jailbreak`
   负责调度模型执行单轮/多轮攻击测试，并将测试过程写入结果文件。
4. `Defense`
   作为可插拔中间层，作用于输入、交互、输出三个阶段。
5. `Analyze`
   负责将原始测试结果转化为判定标签、统计表和图形化输出。
6. `model_registry.py`
   负责统一模型配置解析。
7. `common/`
   负责提供共享配置和运行时抽象。

这种架构的优势在于模块职责清晰：

- `Redteam` 只关心攻击样本生成。
- `Jailbreak` 只关心执行与过程记录。
- `Defense` 只关心拦截、改写和审计。
- `Analyze` 只关心结果判定与可视化。

这使得各模块可以独立演进，同时仍能通过 JSON/CSV/JSONL 形成稳定的数据接口。

## 3. 数据规范化与公共基础设施

在一个实验系统中，最大的工程问题之一并不是“怎么调用模型”，而是“怎么让不同来源的数据、不同供应商的模型接口、不同运行模式共享一套稳定约束”。本项目在这方面有三层公共设计。

### 3.1 数据集规范化

`Jailbreak/jailbreak_tools/loader.py` 提供统一的数据加载器 `Loader`，负责兼容：

- `CSV`
- `JSON`
- `YAML`

不同数据源中的字段可能叫：

- `prompt`
- `text`
- `query`
- `question_zh`

`Loader` 会把它们统一规范为标准字段：

- `id`
- `name`
- `prompt`
- `category`
- `attack_type`

这一设计解决了实验数据来源异构的问题，使后续攻击执行器无需关心原始数据格式。

### 3.2 模型注册与配置解析

项目使用根目录的 `models.yaml` 维护模型信息，`model_registry.py` 负责解析该文件，并统一输出标准模型描述，包括：

- `name`
- `type`
- `provider`
- `base_url`
- `model`
- `api_key`

该模块同时兼容两类模型：

1. 本地 `Ollama`
2. OpenAI-compatible 商业接口

这样做的意义在于：

- 上层模块不必各自解析配置
- 只需要提供模型名，即可解析出具体调用参数
- 本地模型和商用模型可被统一抽象

### 3.3 共享运行时与容错

项目多个模块都体现出一致的运行时设计理念：

- 输出文件实时写盘
- 支持断点续跑
- 支持异步并发
- 支持失败重试或降级回退

例如 `Jailbreak` 多轮执行器中，`MultiTurnResultWriter` 使用异步锁保护文件写入，既保证并发执行，又避免结果文件竞争写入；`Analyze` 中则通过 checkpoint 和 partial rows 实现断点续跑，防止长时间判定过程中意外中断导致全部重算。

从工程角度看，这说明项目并非原型式脚本，而是考虑了实验可重复性与大批量运行稳定性。

## 4. Redteam 模块：攻击提示生成与质量门控制

### 4.1 模块目标

`Redteam` 的任务是从已有攻击样本出发，构造更多变体，从而扩大攻击测试的覆盖面。核心思想是：直接使用原始提示往往不足以全面评估模型，需要借助 surrogate/redteam 机制生成更具迷惑性或更强诱导性的变体。

### 4.2 关键实现位置

主要实现位于：

- `Redteam/redteam_llm/surrogate_model.py`
- `Redteam/redteam_llm/prompting.py`
- `Redteam/redteam_llm/clients.py`
- `Redteam/redteam_convert/redteam_convert.py`

### 4.3 技术原理

`SurrogateModel` 的核心工作流程可以概括为：

1. 加载参考攻击数据集。
2. 根据原始 prompt 检索或抽取可用参考信息。
3. 拼接生成提示词模板，调用 surrogate 模型生成改写结果。
4. 对生成结果进行质量打分，过滤低质量变体。
5. 将最终结果输出为 JSONL，供后续转换与测试。

这里的关键不是“让模型随便改写”，而是通过质量门控制生成结果的有效性。项目在 `surrogate_model.py` 中设计了多种分数：

- `intent`：意图重合度
- `anchor`：锚点覆盖度
- `template`：模板相关性
- `final`：综合得分

综合得分采用加权方式计算，说明系统希望在“保持原始攻击意图”与“形成新表达变体”之间取得平衡，而不是简单同义改写。

### 4.4 实现思路

`Redteam` 采用的不是端到端黑盒生成，而是“模板 + 参考样本 + 质量门”三段式设计。

这种设计的优点是：

1. 可控性强
   可以通过模板约束输出风格。
2. 可解释性好
   质量分数能够解释为什么某些变体被保留。
3. 实验可复用
   同一套模板可以针对不同数据集复用。

### 4.5 转换层

生成后的结果并不直接送去测试，而是通过 `Redteam/redteam_convert/redteam_convert.py` 转换成统一攻击数据集格式，写入 `Attack_Dataset/`。这一步的工程意义是把“生成阶段”和“执行阶段”解耦，从而保证实验流程可回放、可重复。

## 5. Jailbreak 模块：攻击执行与多轮状态机

### 5.1 模块目标

`Jailbreak` 是整个系统的执行核心，负责把攻击提示真正发送给目标模型，并记录模型回复、判定结果和过程状态。

它支持两种主要场景：

1. 单数据集多轮测试
2. 多数据集多模型多轮测试

虽然目录中使用了 `single_jail` 和 `multi_jail` 命名，但从当前实现看，单轮执行器本质上也是围绕多轮对话机制设计，只是调度范围不同。

### 5.2 关键实现位置

主要实现位于：

- `Jailbreak/jailbreak_tools/single_jail/single_jail.py`
- `Jailbreak/jailbreak_tools/multi_jail/multi_jail.py`
- `Jailbreak/jailbreak_tools/multi_jail/runner.py`
- `Jailbreak/jailbreak_tools/multi_jail/adaptive_strategy.py`
- `Jailbreak/jailbreak_tools/multi_jail/result_writer.py`
- `Jailbreak/jailbreak_tools/single_jail/judgers.py`
- `Jailbreak/jailbreak_tools/single_jail/model_tester.py`

### 5.3 多轮执行的核心思想

项目的多轮攻击不是简单重复追问，而是引入了显式的“计划生成”和“动态重规划”机制。

`MultiTurnCaseRunner` 是多轮执行的核心状态机。它维护：

- 当前轮次
- 当前 prompt
- 对话历史
- 目标模型回复
- judge 结果
- 是否触发防御
- 是否需要继续追问

一条样本的执行流程可抽象为：

1. 先由 `AttackPlanner` 为多轮攻击生成初始计划。
2. 每轮把当前 prompt 发送给目标模型。
3. 读取回复后，交给 judge 判定当前轮是否成功。
4. 若未成功，则由 `AdaptivePlannerStrategy` 生成下一轮追问。
5. 若连续拒绝，且上下文仍对齐原目标，则触发 `build_remaining_plan()` 进行后续轮次重规划。

这种机制的技术意义在于，系统不再把多轮攻击视作固定模板循环，而是视作“带反馈的自适应攻击过程”。

### 5.4 攻击规划与追问生成

`AttackPlanner` 负责生成结构化攻击计划，要求模型返回 JSON 形式的 `plan_rounds`，每一轮包含：

- `round_idx`
- `goal`
- `strategy`
- `prompt_candidate`
- `fallback_hint`

如果返回内容无法解析或为空，则系统会回退到 fallback plan。这是一种典型的“结构化生成 + 失败降级”设计。

`AdaptivePlannerStrategy` 负责在上一轮失败后生成下一轮 prompt。它会把以下信息交给辅助模型：

- 原始攻击目标
- 上一轮回复
- 当前轮次
- 对话历史
- judge 状态与原因
- 防御是否启用、采取了什么动作

这说明该系统中的追问生成不是静态脚本，而是上下文敏感的策略生成器。

### 5.5 判定器设计

`Jailbreak` 内部使用 `LayeredJudge` 等判定器，承担在线判定功能。其作用不是做最终学术统计，而是为了在执行过程中判断：

- 当前轮是否成功
- 失败原因是什么
- 下一轮应如何调整策略

因此，`Jailbreak` 内部 judge 更偏向“控制循环”的运行时判定，而 `Analyze` 模块则更偏向“结果归一化与统计”的离线判定。

### 5.6 结果写入与可恢复执行

`MultiTurnResultWriter` 采用 JSONL 实时写盘，并额外支持 debug store：

- 标准结果文件保留摘要信息
- debug store 保留完整原始结构

同时，它支持从已有输出中加载 `(model_name, test_id)` 已完成对，从而在断点续跑时跳过已完成样本。

从实验工程角度看，这种设计非常重要，因为多轮 LLM 实验往往耗时长、失败率高，如果没有断点恢复与实时写盘机制，实验成本会显著上升。

## 6. Defense 模块：三层可插拔防御架构

### 6.1 模块目标

`Defense` 模块的作用不是独立运行，而是作为中间层插入 `Jailbreak` 执行过程中，对 prompt、交互过程和模型输出做风险控制与审计。

它的设计目标是验证三个问题：

1. 是否可以在输入阶段提前发现高风险请求。
2. 是否可以在多轮诱导过程中及时截断危险对话。
3. 是否可以在输出阶段对有害内容做替换、脱敏或归档。

### 6.2 核心架构

`DefenseEngine` 是统一调度器，接收三个可选模块：

- `input_module`
- `interaction_module`
- `output_module`

引擎内部通过 `_ACTION_PRIORITY` 定义动作强弱顺序：

- `allow`
- `rewrite`
- `redact`
- `truncate`
- `replace`
- `block`

这意味着多层决策不是简单覆盖，而是有“更强动作优先”的聚合逻辑。

### 6.3 输入层防御

`InputDefenseModule` 的核心思路是“规则检测 + 关键词风险分类”。

其输入风险分数来自两部分：

1. 规则命中分数
   例如 prompt injection、角色越权、规则规避等模式。
2. 关键词风险分类器分数
   `KeywordRiskClassifier` 会检测例如：
   - `如何`
   - `绕过`
   - `越狱`
   - `破解`
   - `恶意软件`
   - `赌博`

然后根据总分触发三种动作：

- 高风险：`block`
- 中风险：`rewrite`
- 低风险：`allow`

其中 `rewrite` 会把原始请求包裹成安全改写版本，让模型优先以安全模式回应。

### 6.4 交互层防御

`InteractionDefenseModule` 主要针对多轮诱导问题。

它考虑两个因素：

1. 当前风险分数
2. 当前轮次

如果满足：

- 风险高到阈值
- 或者已经进入多轮诱导并出现危险信号
- 或者超过允许轮次上限

则会返回：

- `block`
- 或 `truncate`

这说明交互层不是只看单轮内容，而是把“轮次演化”纳入决策，这一点对多轮越狱防御非常关键。

### 6.5 输出层防御

`OutputDefenseModule` 对模型回复做后处理。

其核心设计包括：

1. 对输出文本做风险规则检测。
2. 根据上下文状态做风险加权。
3. 对中风险内容执行 `redact`。
4. 对高风险内容执行 `replace`。
5. 把原始回复和处理结果归档到审计存储中。

这里的一个亮点是输出层会结合上下文做额外加权，例如：

- 如果交互状态已经是 `warning` 或 `restricted`
- 如果处于多轮后续轮次
- 如果已有角色操控、规则豁免等越狱信号

则输出风险分数会被抬高。也就是说，该模块不是孤立地看文本，而是把对话上下文纳入输出风险判断。

### 6.6 技术评价

`Defense` 模块的价值不在于当前规则是否已经“完美”，而在于它已经形成了一套可扩展的防御框架：

- 每层都遵循统一接口
- 每层都返回标准化 `DefenseDecision`
- 决策具有风险分数、原因、改写文本、审计载荷

这为后续接入更复杂的模型型防御器、分类器或审计存储提供了良好基础。

## 7. Analyze 模块：离线判定、统计与可视化

### 7.1 模块目标

`Analyze` 的任务是将 `Jailbreak` 产生的原始 JSONL 结果转化为研究可用的统计结论。

它负责：

1. 读取结果文件。
2. 对每条结果做统一判定。
3. 输出逐条明细和聚合表。
4. 生成图表，支持结果汇报。

### 7.2 关键实现位置

主要实现位于：

- `Analyze/cli.py`
- `Analyze/pipeline.py`
- `Analyze/multi_turn/cli.py`
- `Analyze/multi_turn/pipeline.py`
- `Analyze/judges/final_judge.py`
- `Analyze/stats.py`
- `Analyze/plotting.py`

### 7.3 判定体系设计

`Analyze` 中的判定以 `JudgeDecision` 为标准输出结构，包含：

- `is_jailbreak`
- `risk_level`
- `evidence_spans`
- `reasoning_tags`
- `judge_source`

当前静态实验判定器为：

1. `FinalJudge`

这种分层设计的意义在于：

- 最终输出结构仍保持统一
- 可以针对固定单轮结果文件做论文导向分析

### 7.4 最终判定模式

`paper` 模式针对固定格式单轮结果文件，联合分析请求与响应内容，适合论文正式实验统计。

`paper` 模式会直接输出危险越狱率、风险等级分布、不确定率、高风险占比和代表性案例表。

从研究方法角度看，这是一种“后验约束”设计，目的是降低离线统计中的伪阳性或结构性噪声。

### 7.6 单轮与多轮分析

单轮分析读取普通结果文件，围绕每条样本的最终回复做判定。

多轮分析除了抽取选中的回复文本做最终判定外，还会对每轮回复做同一套分析模块重评，并基于重评结果生成：

- `multi_turn_round_metrics.csv`
- `multi_turn_cumulative_success.png`
- `multi_turn_first_success_distribution.png`

这使得系统不只回答“最终是否成功”，还回答“第几轮首次成功”“增加轮次是否显著提升成功率”等问题。

### 7.7 统计与图表

`stats.py` 负责生成聚合指标，例如：

- `success_rate`
- `uncertain_rate`
- `risk_0_ratio` 到 `risk_4_ratio`
- `95% Wilson CI`

`plotting.py` 负责生成多种图表：

- 越狱成功率图
- 风险分布图
- 不确定性概览图
- 风险热力图
- 多轮累计成功率图
- 多轮首次成功轮次分布图

从汇报角度看，`Analyze` 模块实现了从原始日志到可解释结果的完整闭环，是支撑实验结论输出的核心模块。

## 8. 交互式入口与系统可用性设计

除了各模块内部实现，本项目还通过 `Jelly_Z/bin/` 下的交互式脚本把实验流程组织成可操作工具，例如：

- `redteam`
- `convert`
- `jailbreak`
- `analyze`

这些脚本负责：

- 枚举可选输入
- 提示用户选择模型与数据集
- 检查环境与依赖
- 自动拼接命令行参数
- 对错误进行屏蔽或重试

例如 `Jelly_Z/bin/analyze` 已经具备：

- 自动识别单轮/多轮结果文件
- 混选报错
- 多轮结果自动切到 `Analyze.multi_turn.cli`
- 失败时对 trace 做过滤，并在需要时断点续跑

这类入口设计使得系统既适合代码级开发，也适合实验人员直接使用。

## 9. 项目的关键工程亮点

从技术实现角度，我认为这个项目有以下几个工程亮点。

### 9.1 模块边界清晰

攻击生成、攻击执行、防御决策、结果分析彼此解耦，既方便独立实验，也方便替换单个模块。

### 9.2 数据接口统一

通过 `Loader`、`model_registry`、`JudgeDecision` 等标准化结构，项目显著降低了模块耦合。

### 9.3 面向实验场景的稳定性设计

项目大量采用：

- JSONL 增量写盘
- async 并发
- checkpoint/resume
- fallback plan
- 输出归档

这些机制对大规模 LLM 实验尤为重要。

### 9.4 多轮攻击不是机械循环，而是反馈驱动

`AttackPlanner + AdaptivePlannerStrategy + Judge` 构成了闭环，使多轮攻击具备一定自适应能力，这是项目技术深度的重要体现。

### 9.5 防御框架可扩展

三层防御并非硬编码在执行逻辑中，而是通过统一接口插入，具有明显的研究扩展空间。

### 9.6 分析层具备研究输出能力

`Analyze` 模块不是简单统计脚本，而是形成了“判定器 + 结构约束 + 聚合统计 + 可视化”的完整研究支持工具链。

## 10. 当前局限与后续改进方向

尽管该项目已经形成完整链路，但从技术汇报角度，也需要明确其局限性。

### 10.1 规则型模块仍偏启发式

无论是输入层风险分类器、输出层规则，还是 `Analyze` 的关键词判定器，目前仍较依赖人工规则，存在：

- 泛化能力有限
- 对新型攻击样式敏感度不足
- 易受 prompt 包装方式影响

### 10.2 多轮成功标签与文本判定可能不一致

项目中存在“运行时 judge 标记为 refused，但文本内容已经泄露明显危险信息”的情况。这说明在线判定与离线判定之间仍可能出现语义偏差，后续可以进一步设计一致性校验机制。

### 10.3 聚合维度仍较粗

当前 `Analyze` 聚合主要按 `source_file` 进行，若一个文件内部混合多种模型、攻击方式或策略，结果解释会受到限制。后续可扩展为多键聚合，例如：

- `model_name`
- `attack_type`
- `category`
- `source_file`

### 10.4 防御策略仍偏规则化

虽然框架已经成型，但当前防御层尚可进一步接入：

- 模型型分类器
- 语义安全判断器
- 基于历史轨迹的风险累积机制

### 10.5 结果与实验管理还可进一步产品化

目前项目已经支持文件级归档，但若继续发展，可增加：

- 实验元数据数据库
- Web 展示面板
- 模型对比报表自动生成
- 批量实验调度器

## 11. 结论

总体来看，`Experiment` 项目已经形成了一套结构完整的 LLM 安全评测系统，其价值不只是“能跑通实验”，而在于它具备以下技术特征：

1. 以完整链路组织实验，而非孤立脚本。
2. 兼顾研究灵活性与工程稳定性。
3. 支持多轮攻击、自适应追问与可插拔防御。
4. 将原始运行日志转化为可解释、可汇报的结构化分析结果。

如果用于向导师汇报，这个项目最值得强调的并不是单个函数实现，而是它已经把“攻击生成、攻击执行、防御验证、结果分析”整合为一个统一实验平台。换言之，本项目的技术贡献体现在系统化设计、模块化实现和研究流程闭环三个层面。
